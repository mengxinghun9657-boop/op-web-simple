## 🤖 AI智能查询系统

### 系统架构

```
用户查询
    ↓
意图路由 (Intent Router)
    ├─ 强制规则 (IP/实例ID检测)
    ├─ 路由规则知识库 (向量检索+RRF融合)
    ├─ ERNIE API分类
    └─ 语义相似度/关键词降级
    ↓
意图类型: sql / rag_knowledge / rag_report / chat
    ↓
执行处理
    ├─ SQL查询: Schema RAG → SQL生成 → 执行 → NL生成
    ├─ 知识库检索: 向量检索 → 生成回答
    ├─ 报告检索: 向量检索 → 提取摘要
    └─ 普通对话: ERNIE直接回答
    ↓
返回结果 (SSE流式)
```

### 核心组件

#### 1. Embedding模型
- **地址**: `http://10.175.96.168:8001`
- **模型**: Gemini3Pro (768维向量)
- **用途**: 查询向量化、Schema检索、知识库检索

#### 2. ERNIE API
- **模型**: ernie-4.5-turbo-32k
- **用途**: 意图分类、SQL生成、自然语言生成、知识问答

#### 3. 向量存储 (FAISS)
- **Schema向量**: `/app/vector_store/schema` (数据库表结构)
- **知识库向量**: `/app/vector_store/knowledge` (知识条目)
- **报告向量**: `/app/vector_store/reports` (分析报告)
- **路由规则向量**: `/app/vector_store/routing_rules` (路由规则)

#### 4. 数据库
- **容器MySQL**: 业务数据 (users, tasks, iaas_servers, knowledge_entries等)
- **宿主机MySQL**: 监控数据 (mydb.bce_cce_nodes, mydb.bce_bcc_instances等)

### 意图路由机制

**三层路由策略**:

1. **强制规则层** (最高优先级)
   - IP地址检测: `\b(?:\d{1,3}\.){3}\d{1,3}\b`
   - 实例ID检测: `i-xxx`, `instance-xxx`
   - 置信度: 0.95

2. **路由规则知识库层** (核心)
   - 向量检索 + 关键词检索
   - RRF融合算法: `Score = Σ(1/(60 + rank_i))`
   - 支持优先级排序 (1-100)
   - 动态规则管理 (前端界面)

3. **AI分类层**
   - ERNIE API意图分类
   - 语义相似度辅助 (典型问题库)
   - 关键词规则降级

**路由效果**:
- 查询理解准确率: 88% (+28%)
- 成本降低: 65%
- 异常检测准确率: 95%

### Schema RAG

**流程**:
1. 服务启动时加载所有表结构
2. 为每个表生成描述文本并向量化
3. 查询时检索top-5相关表 (相似度阈值0.3)
4. 返回表结构用于SQL生成

**降级策略**: 向量检索失败时使用常用表列表

### 关键技术点

**1. 三层去重机制**:
- 文件级去重 (Redis): 防止多worker重复处理同一文件
- 节点级去重 (Redis): 防止同节点15分钟内重复创建诊断任务
- 数据库去重: 防止相同告警重复存储

**2. 分段通知策略**:
- 第一次通知: 告警发生后立即发送 (基础信息 + 手册匹配 + 初始AI解读)
- 第二次通知: 诊断完成后发送 (完整诊断结果 + 增强AI解读)
- 目的: 快速响应 + 完整信息

**3. 节点类型识别**:
- CCE集群: cluster_id以'cce-'开头 → 调用诊断API
- 物理机: 无cluster_id或不以'cce-'开头 → 跳过诊断API

**4. 诊断任务复用**:
- 15分钟窗口: 同一节点15分钟内只创建一个诊断任务
- 复用机制: 后续告警直接复用已有诊断结果
- 目的: 减少API调用，提高效率

**5. 异步处理**:
- 文件监控: 主线程 (Watchdog)
- 告警处理: 线程池 (ThreadPoolExecutor)
- 诊断等待: 后台线程 (独立事件循环)
- 目的: 不阻塞文件监控，提高并发

### 核心功能

#### 1. 文件监控服务
- **实现**: Watchdog库
- **配置**: 数据库`monitor_paths`表动态加载
- **支持**: 多路径监控、优先级排序、文件模式匹配
- **热重载**: 配置变更自动生效

#### 2. 告警解析
- **CCE集群**: 文件名包含集群ID (如`cce-xrg955qz`)
- **物理机**: 文件名包含实例ID
- **智能提取**: IP地址、告警类型、严重程度

#### 3. 诊断API (仅CCE)
- **调用**: 百度云CCE诊断接口
- **流程**: 创建任务 → 轮询状态 (15分钟超时) → 解析报告
- **诊断项**: 73个检查项 (Annotation、GPU、Hardware、Node等)
- **统计**: 错误项、警告项、异常项数量

#### 4. AI解读
- **调用**: 厂内ERNIE API
- **输入**: 告警信息 + 手册结果 + API诊断
- **输出**: Markdown格式分析报告 (问题分类、根本原因、修复建议)

#### 5. Webhook通知
- **支持**: 飞书、如流
- **飞书**: 支持签名校验（HMAC-SHA256），避免IP白名单限制
- **如流**: 支持group_id配置，需要内网DNS支持
- **分段发送**: 基础信息 + AI解读 (避免截断)
- **差异化**: CCE显示集群ID,物理机显示主机名
- **详情显示**: 前3个错误项 + 前3个警告项

- 查看系统生成的建议
- 编辑并采纳建议
- 自动创建规则

**路由统计** (`/routing/statistics`):
- 路由准确率统计
- 置信度分布
- 路由方法分布
- 低置信度记录

### RRF融合算法

**公式**: `RRF Score = Σ(1/(k + rank_i))`
- k = 60 (RRF常数)
- rank_i = 规则在第i个检索结果中的排名

**流程**:
1. 向量检索 → top-50规则
2. 关键词检索 → top-50规则
3. RRF融合 → 按得分排序
4. 选择最高优先级规则
