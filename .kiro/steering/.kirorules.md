## 1. 双重记忆协议 (Dual Memory Protocol)
| 类型 | 文件 | 用途 | 更新时机 |
|------|------|------|----------|
| [ROM] | `功能逻辑.md` | 项目架构、API 接口、服务器信息 | 大功能测试完毕后 |
| [RAM] | `PROJECT_STATUS.md` | **仅记录当前活跃问题** | 每次任务状态变化时 |

### 📝 RAM 记录规范（强制）
- **格式**：`[状态] 问题简述 | 根因 | 更新时间`
- **长度**：每问题 ≤ 2 行，总文件 ≤ 50 行
- **清理**：用户确认解决后 **立即删除** 该条目
- **归档**：重大修复可写入 `CHANGELOG.md`（可选）

### 🚫 绝对禁止
- 禁止记录历史修复详情（超过 24 小时的问题）
- 禁止在 PROJECT_STATUS.md 中写入完整代码
- 禁止保留已验证解决的问题
- 禁止创建 `SUMMARY.md`, `REPORT.md`, `ANALYSIS.md` 等临时报告

## 2. 生命周期程序 (Lifecycle Hooks)
### 每次接收新指令时：
1. 读取 `PROJECT_STATUS.md`
2. **检查是否有超过 24 小时的已解决问题** → 移动到"已完成"或删除
3. 更新当前任务状态
4. 响应格式：`"根据 PROJECT_STATUS.md，当前活跃问题 [X] 个..."`

### 每次修复验证通过后：
1. 更新对应问题状态为 ✅
2. **等待用户确认** → 确认后删除该条目
3. 可选：写入 `CHANGELOG.md` 归档

## 3. 行为红线 (Hard Constraints)
- **拒绝盲猜**：修复报错前必须看到 Log (`tail -n 50 logs/app.log`)。没有 Log 不准改代码
- **最小化修改**：所有后端逻辑必须加 `try-except` 和 `logger.error`
- **Token 节约**：只读必要代码（使用 grep 或行号），Python 强制使用 `python3`
- **简洁沟通**：不输出"正如我之前分析的"等废话，直接给代码

## 7. 基础架构变更影响面分析 (Infrastructure Change Impact)
### 🚨 强制检查清单
修改基础架构代码（axios拦截器、API响应格式、数据库Schema、认证逻辑等）时，**必须**执行以下检查：

#### 7.1 API响应格式变更
**触发条件**：修改 `axios.js`、`errorHandler.js`、API响应拦截器
**强制检查**：
```bash
# 1. 搜索所有使用该API的前端组件
grep -r "configApi\|loadConfig\|saveConfig" frontend/src/
grep -r "\.config\|\.data\.config" frontend/src/

# 2. 检查响应数据访问模式
grep -r "res\.config\|response\.config" frontend/src/
grep -r "res\.data\|response\.data" frontend/src/

# 3. 验证统一响应格式兼容性
# 确保所有组件支持 {success, data, message} 格式
```

#### 7.2 数据库Schema变更
**触发条件**：修改数据库表结构、字段类型、约束条件
**强制检查**：
```bash
# 1. 搜索所有使用该表的代码
grep -r "table_name\|TableModel" backend/
grep -r "column_name" backend/

# 2. 检查ORM模型定义
find backend/ -name "*.py" -exec grep -l "class.*Model" {} \;

# 3. 验证迁移脚本兼容性
```

#### 7.3 认证/权限逻辑变更
**触发条件**：修改JWT、权限检查、用户角色逻辑
**强制检查**：
```bash
# 1. 搜索所有权限检查点
grep -r "isAdmin\|role\|permission" frontend/src/
grep -r "@require_role\|check_permission" backend/

# 2. 检查前端路由守卫
grep -r "router\.beforeEach\|meta\.requiresAuth" frontend/src/
```

### 7.7 经验教训记录
**案例1**: 修复登录问题时统一axios响应格式 (2026-03-05)
- **问题**: 删除errorHandler.js重复拦截器，统一使用axios.js的{success,data,message}格式
- **遗漏**: 监控配置和分析配置组件仍使用旧的res.config访问方式
- **教训**: API响应格式变更必须全面检查所有使用该格式的前端组件
- **预防**: 使用grep搜索所有相关组件，添加兼容性处理

**案例2**: iCafe连接测试显示假成功问题 (2026-03-11)
- **问题**: 前端测试连接使用`configApi.testICafeConnection()`但未正确导入函数
- **根因**: 导入语句缺少`testICafeConnection`，调用时使用了不存在的`configApi`对象
- **教训**: 前端组件调用API函数时必须检查导入语句和函数名称的一致性
- **预防**: 
  1. 使用IDE的自动导入功能避免手动输入错误
  2. 调用API前先检查函数是否正确导入
  3. 测试功能时必须验证真实的网络请求，不能依赖模拟逻辑

**记住**: 基础架构是所有功能的根基，任何变更都可能产生连锁反应。宁可多检查，不可漏检查。
