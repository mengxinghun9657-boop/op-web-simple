## 🚨 硬件告警诊断系统

### 系统概述

自动监控告警文件、解析告警信息、匹配故障手册、调用CCE诊断API、生成AI解读并发送Webhook通知。

### 核心流程

```
文件监控 → 告警解析 → 手册匹配 → AI解读 → Webhook通知
                ↓
         (CCE集群) → 诊断API → 后台线程等待 → 增强AI解读 → 二次通知
```

### 关键功能

**1. 文件监控**
- 监控目录: `/app/alerts_source` (宿主机只读挂载)
- 监控模式: PollingObserver (轮询避免inotify限制)
- 配置管理: 数据库`monitor_paths`表动态配置
- 去重机制: Redis文件级锁 (TTL 1小时)

**2. 告警解析**
- 支持多行JSON数组格式
- 自动识别CCE集群 (cluster_id以'cce-'开头) 和物理机
- 提取: alert_type, component, severity, ip, cluster_id

**3. 故障手册匹配**
- 数据来源: `fault_manuals`表 (290条记录，CSV导入)
- 匹配规则: alert_type + component
- 返回: 故障名称、影响、恢复方案、解决方案

**4. CCE诊断API (仅CCE集群)**
- 调用: 百度云CCE节点诊断接口
- 诊断项: 73个检查项 (Annotation/GPU/Hardware/Node等)
- 后台线程: 独立事件循环，15秒轮询，最大等待15分钟
- 复用机制: 15分钟窗口内同节点复用诊断结果

**5. AI解读**
- 模型: 厂内ERNIE 4.5 Turbo 32K
- 输入: 告警信息 + 手册匹配 + (可选)API诊断
- 输出: Markdown格式分析报告

**6. Webhook通知**
- 支持: 飞书 (HMAC-SHA256签名) / 如流 (group_id)
- 分段发送: 基础信息 + AI解读 (避免消息截断)
- 过滤: severity_filter, component_filter

### 数据库设计

**alert_records**: 告警记录 (alert_type, ip, cluster_id, severity, timestamp, raw_data)
**diagnosis_results**: 诊断结果 (manual_matched, api_task_id, api_diagnosis, ai_interpretation, notified)
**fault_manuals**: 故障手册 (name_zh, alert_type, component, solution, impact, recovery)
**webhook_configs**: Webhook配置 (type, url, secret/group_id, severity_filter)
**monitor_paths**: 监控路径配置 (path, file_pattern, priority, enabled)

### 前端功能

**告警列表** (`/alerts/list`):
- 查询、排序、筛选
- 批量选择、补发通知
- 快速创建iCafe卡片
- 编辑告警字段 (cluster_id, component, severity等)

**告警详情** (`/alerts/:id`):
- 告警基础信息
- 手册匹配结果
- CCE诊断报告 (73项详情)
- AI解读报告
- 编辑和重新诊断功能

**配置管理** (`/system-config`):
- 监控路径配置
- Webhook配置
- iCafe配置

### 关键技术点

**1. 重新诊断功能** (2026-03-12修复)
- 用户可手动修正cluster_id等字段
- 点击"重新诊断"删除旧诊断结果，重新执行完整流程
- 使用主键重新查询避免Session缓存问题

**2. 后台线程诊断** (2026-03-12修复)
- 使用`logger.warning()`确保生产环境日志可见
- 独立Session和事件循环避免阻塞
- 使用`flush()+commit()`确保数据库更新成功

**3. 三层去重机制**
- 文件级: Redis SETNX锁
- 节点级: Redis诊断锁 (15分钟窗口)
- 数据库级: alert_type + ip + timestamp唯一性检查

## 🤖 AI智能查询系统

### 系统架构

```
用户查询 → 意图路由 → 执行处理 → 返回结果(SSE流式)
```

### 核心组件

**1. Embedding模型**
- 地址: `http://10.175.96.168:8001`
- 模型: Gemini3Pro (768维向量)

**2. ERNIE API**
- 模型: ernie-4.5-turbo-32k
- 用途: 意图分类、SQL生成、自然语言生成

**3. 向量存储 (FAISS)**
- Schema向量: 数据库表结构
- 知识库向量: 知识条目
- 报告向量: 分析报告

### 意图路由

**三层策略**:
1. 强制规则: IP地址/实例ID检测 (置信度0.95)
2. 路由规则知识库: 向量+关键词检索，RRF融合
3. AI分类: ERNIE API意图分类

**意图类型**: sql / rag_knowledge / rag_report / chat

### Schema RAG

- 服务启动时加载所有表结构并向量化
- 查询时检索top-5相关表 (相似度阈值0.3)
- 返回表结构用于SQL生成

### 5. 内网部署故障手册导入失败 ✅ 已修复（v4 - 降低worker数量）
**问题**:
```
ERROR: 数据库连接失败（已重试5次）: (pymysql.err.OperationalError) (2003, "Can't connect to MySQL server on 'mysql' ([Errno 111] Connection refused)")
./start.sh: line 375: syntax error near unexpected token `('
cluster-backend | 2026-03-13 19:07:16 | WARNING | ⚠️ 数据库初始化失败（尝试 8/10）（日志重复多次）
```

**原因**:
1. **SQLAlchemy 2.0 兼容性**: `db.execute('SELECT 1')` 需要改为 `db.execute(text('SELECT 1'))`
2. **Shell 脚本字符串问题**:
   - Python f-string 中的单引号被 shell 双引号误解析
   - 导致语法错误
3. **MySQL 连接池被占满**: 导入 292 条记录后，MySQL 需要时间处理写入，连接池短暂不可用
4. **多 worker 并发连接**: 4 个 worker 同时重试连接（4 × 10 次 = 40 个并发连接），耗尽 MySQL 连接池（默认 151 个）

**修复内容 v1** (pack-offline.sh 行 437-504):
```python
# 修复前（双引号包裹，f-string 导致 shell 语法错误）
docker ... exec -T backend python3 -c "
logger.warning(f'数据库连接失败（尝试 {attempt+1}/5）: {e}')
"

# 修复后（单引号包裹，使用 % 格式化，导入 text()）
docker ... exec -T backend python3 -c '
from sqlalchemy import text
db.execute(text("SELECT 1"))
logger.warning("数据库连接失败（尝试 %d/5）: %s" % (attempt+1, e))
'
```

**修复内容 v2** (pack-offline.sh 行 436-513):
```bash
# 增加导入后等待时间
sleep 20  # 从 3 秒增加到 20 秒，给 MySQL 足够时间处理 292 条记录的写入

# 改进错误处理（失败不阻塞部署）
```

**技术要点**:
- Shell 中的 Python 内联代码应使用**单引号**包裹（避免变量替换）
- Python 字符串格式化改用 `%` 而非 f-string（避免 shell 误解析 `{}`）
- SQLAlchemy 2.0 的原始 SQL 必须用 `text()` 包装

- **降级策略确保用户始终有解决路径**
- **多 worker 启动时应降低并发连接数**，避免耗尽 MySQL 连接池（gunicorn lifespan 每个 worker 独立执行）
